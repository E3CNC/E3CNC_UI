# v0.8.0 — Meet the New E3CNC

**Date:** 2026-06-26

v0.8.0 rewrites how E3CNC is deployed. The old model scattered files across half a dozen runtime directories with no rollback and no safety net. This release consolidates everything into a single stack artifact built in CI, applied with one command, and guarded by health checks that auto-rollback on failure. The repo was renamed from `E3CNC_UI` to `E3CNC` to reflect that this is a full control stack, not just a dashboard. Key highlights: atomic staged releases at `~/e3cnc/releases/`, a Moonraker API that lets you update from the frontend, new CLI commands for release management, and first-class multi-instance support.

## Why upgrade?

| Before v0.8.0 | v0.8.0 |
|---|---|
| Multiple manual copy steps | One command: `./e3cnc-cli update` |
| Partial updates if something fails | Atomic activation with automatic rollback |
| No rollback — you're on your own | `./e3cnc-cli rollback` reverts everything |
| Built frontend on the machine (slow, OOM on 32-bit) | Pre-built artifact from CI, just download |
| No health checks — broken updates go live | 7 checks, auto-rollback on failure |
| SSH-only updates | Update from the frontend UI or CLI |
| Had to track E3CNC_UI (not just a UI!) | Just `E3CNC` — the name matches the project |

v0.8.0 completely rethinks how E3CNC updates work. The old way meant running a handful of unrelated copy commands that could leave your machine in a half-baked state. The new way is one command, one artifact, one happy path.

## What changed

**The repo got a new name and a cleaner layout.** `E3CNC_UI` is now just `E3CNC` — because this project grew way beyond "just a UI." The internal `E3CNC/` directory was flattened to the repo root, deploy scripts moved to `build-scripts/`, Moonraker source moved to `moonraker/`, and the old `E3CNC/moonraker-mcp/` is history.

**Updates now ship as a single artifact.** CI builds `e3cnc-stack-v0.8.0.tar.zst` — a compressed bundle containing the frontend, Moonraker components, Klipper extras, macros, and metadata. Your machine never builds anything. It downloads, checks the SHA-256, extracts, and activates. That's it.

**Releases are staged and rollback is built in.** Every update lands in `~/e3cnc/releases/v0.8.0/`. A symlink (`current → releases/v0.8.0`) switches between them atomically. If health checks fail after an update, the CLI rolls you back automatically. No more "oops, I'll fix it manually."

**The CLI grew up.** New commands:

```
e3cnc-cli update         # full stack update (14 steps, one command)
e3cnc-cli releases       # what's installed
e3cnc-cli rollback       # oops, take me back
e3cnc-cli prune          # clean up old releases
e3cnc-cli migrate        # move from the old layout to the new one
e3cnc-cli instances      # see all your instances at a glance
```

**You can update from the frontend now.** Click the power button in the top-right corner, hit **Update Stack**, and watch a spinner while Moonraker runs the update in the background. Green toast = done. Red toast = check the logs.

**Multi-instance hosts are supported.** Shared release payload, per-instance services and ports. We even set up a virtual test instance on port 7135 to prove it works.

## What stayed the same

Klipper is still upstream. Moonraker isn't forked — only component files live in the repo. The Ansible playbooks still work if you prefer them. And the frontend is still a Mainsail fork at heart.

## Try it

```bash
# If you're on an old install
cd ~/E3CNC
git pull
./e3cnc-cli migrate

# For everyone
./e3cnc-cli update
```
