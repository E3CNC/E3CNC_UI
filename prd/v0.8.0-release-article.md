# E3CNC v0.8.0 — Single-Deploy Architecture

**Date:** 2026-06-26

E3CNC v0.8.0 introduces a completely redesigned deployment model. What started as a Mainsail fork with a few Ansible scripts bolted on has evolved into a self-contained stack with atomic updates, health checks, rollback, and a Moonraker API to drive it all from the frontend.

## The problem

Before v0.8.0, updating E3CNC meant running a series of independent copy operations:

```
git pull               # pull new code
ansible-playbook ...   # vendor agent files into Moonraker
cp ...                 # copy Klipper extras
cp ...                 # copy macros
rsync ...              # deploy frontend
sudo systemctl restart # restart services
```

Each step touched a different runtime tree: `~/moonraker/`, `~/klipper/`, `~/printer_data/`, `~/mainsail/`. If the power died between step 3 and step 4, you had a partial update with no way to roll back.

The repo was named `E3CNC_UI` — implying it was *just a UI* — but it had grown to own Moonraker components, Klipper extras, macros, CLI tooling, and deployment logic all living in an internal `E3CNC/` subdirectory.

## The v0.8.0 solution

### 1. Repo rename and layout flattening

The repo was renamed from `E3CNC/E3CNC_UI` to `E3CNC/E3CNC` and the internal `E3CNC/` directory was flattened to the root:

```
  E3CNC/
-   E3CNC/
-     extras/...         →   extras/
-     macros/...         →   macros/
-     post_processors/   →   post_processors/
-     scripts/           →   scripts/
-     theme.json         →   theme.json
-     moonraker-mcp/     →   moonraker/ (restructured)
-   scripts/             →   build-scripts/
```

Deploy scripts (`post_update.sh`, `install_to_moonraker.sh`) moved to `build-scripts/`, while runtime scripts (`cnc_metadata_extractor.py`) stayed in `scripts/`. The Moonraker component source moved from `E3CNC/moonraker-mcp/src/moonraker_mcp/` to `moonraker/cnc_agent/`, `moonraker/cnc_metadata/`, and `moonraker/mcp/`.

### 2. One artifact, one apply step

Instead of pulling source and building on the target machine, CI now produces one stack artifact:

```
e3cnc-stack-v0.8.0.tar.zst + checksum
  ├── frontend/          # compiled SPA
  ├── moonraker/         # component files + MCP server
  ├── klipper/extras/    # Klipper plugins
  ├── config/macros/     # CNC macros
  ├── scripts/           # metadata extractor
  ├── theme.json         # branding
  └── manifest.json      # version metadata
```

The target machine never builds anything. It downloads the artifact, verifies the SHA-256 checksum, extracts it, and activates it — all in one controlled sequence.

### 3. Staged releases with rollback

Releases are stored as immutable directories under `~/e3cnc/releases/`:

```
~/e3cnc/
  releases/
    v0.7.11/
    v0.8.0/
  current → releases/v0.8.0
  journal.json
```

Activation is an atomic symlink switch:

```bash
ln -sfn releases/v0.8.0 ~/e3cnc/current.new
mv -T ~/e3cnc/current.new ~/e3cnc/current   # atomic
```

If the update fails health checks, the CLI automatically rolls back to the previous release. The journal tracks `current`, `previous`, and `last_known_good` for deterministic recovery.

### 4. Full deploy sequence (14 steps)

The `e3cnc-cli update` command now owns the complete deployment flow:

```
 1. Download artifact (with .part staging)
 2. Verify SHA-256 checksum
 3. Pre-flight checks (Python, disk space)
 4. Back up mutable state (config, DB, WCS offsets)
 5. Extract release into ~/e3cnc/releases/
 6. Install pip dependencies (from vendored wheels)
 7. Run config/schema migrations
 8. Activate (atomic symlink switch)
 9. Sync runtime files to live paths
10. Update systemd drop-in paths
11. Restart services (Moonraker → Klipper)
12. Health checks (7 checks with retry)
13. On success: update journal, prune old releases
14. On failure: automatic rollback + state restore
```

### 5. Moonraker API for frontend-driven updates

The `cnc_agent` Moonraker component now exposes four endpoints:

| Endpoint | Method | Purpose |
|---|---|---|
| `/machine/e3cnc/info` | GET | Version, releases, instance list |
| `/machine/e3cnc/update` | POST | Trigger full stack update (background task) |
| `/machine/e3cnc/rollback` | POST | Roll back to previous release |
| `/machine/e3cnc/releases` | GET | List installed releases |

The frontend's top-right menu now shows the current instance info and has an **E3CNC** section with **Update Stack** and **Rollback** buttons. Clicking **Update Stack** shows a full-screen overlay spinner while the update runs in the background, then displays a success/failure toast.

### 6. New CLI commands

```
e3cnc-cli update              # full 14-step stack update
e3cnc-cli releases            # list installed releases
e3cnc-cli rollback [version]  # revert with full state recovery
e3cnc-cli prune [--keep N]    # GC old releases
e3cnc-cli migrate             # migrate from old layout
e3cnc-cli instances           # list instances with ports and URLs
```

All legacy commands (`install`, `deploy`, `status`, `backup`, `restore`, `diagnose`, `logs`) remain available.

### 7. Multi-instance support

The architecture supports multiple Klipper/Moonraker instances on one host. The release payload is shared, but activation is instance-aware — each instance gets its own service names, ports, web roots, and config directories.

A parallel test instance (`test2`) with a virtual printer was set up on port 7135 to validate this.

## Migration path

Existing installations can migrate via:

```bash
./e3cnc-cli migrate
```

This downloads the latest stack artifact, creates the `~/e3cnc/` layout, extracts and activates the release, and rewrites `[update_manager E3CNC_UI]` entries to `[update_manager E3CNC]`.

## What didn't change

- **Klipper** remains mostly upstream — only the `work_coordinate_systems.py` extras plugin is maintained
- **Moonraker** is not forked — only component files live in the monorepo
- **The frontend** is still a Mainsail fork at its core — the Vue 3 / Vuetify 3 stack is unchanged
- **Ansible playbooks** still exist for legacy installs but are no longer the primary deployment path

## Credits

This release was driven by a design review of the deployment model that identified gaps in partial-update safety, rollback capability, and frontend-backend version coupling. The v0.8.0 architecture addresses all of these with a single-artifact, single-apply-step model.
